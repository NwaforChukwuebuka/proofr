## Milestone 6 â€” Revenue engine

- `GET /api/merchants/:id/revenue` implemented in [app/api/merchants/[id]/revenue/route.ts](app/api/merchants/[id]/revenue/route.ts). Full request/response/auth detail in [api.md](api.md).
- **"Gross inflow" vs "verified revenue" definition** (PRD names both, doesn't define the gap): for this milestone they are **identical** â€” both are `sum(transactions.amount)` for the merchant. Reasoning: "verified" is meant to mean "with fraud-flagged activity excluded," but the fraud rule engine (milestone 8) doesn't exist yet and `fraud_flags` is empty, so there is nothing to exclude today. Making `verifiedRevenue` a different number now (e.g. some ad-hoc discount) would just be an arbitrary number, not a real verification signal. **Milestone 8 must revisit this**: once fraud rules populate `fraud_flags`, `verifiedRevenue` should become `sum(amount) for transactions with no open high/medium-severity flag` (exact threshold is milestone 8's call, see `fraud-rules.md`) while `grossInflow` stays the unfiltered sum â€” that's the real distinction the PRD is gesturing at.
- **Gross vs net `amount` decision** (seam flagged by milestone 5): both `grossInflow` and `verifiedRevenue` use the first-class `transactions.amount` column (Monnify's gross `amountPaid`), not the fee-adjusted `settlementAmount` buried in `raw_payload->'eventData'->>'settlementAmount'`. Reasoning: `amount` is indexed, typed, and always present; `settlementAmount` is an unindexed JSON path that would need a migration to promote to a first-class column to aggregate cheaply and correctly, and Monnify's fee is a processing-cost detail, not a fraud/authenticity signal â€” conflating "fee-adjusted" with "verified" would misuse the word the PRD is careful to use for fraud screening. If a later milestone wants a net/settlement figure (e.g. "what actually lands in the merchant's bank"), it should be a third, separately-named field, not a redefinition of `verifiedRevenue`.
- **Auth implemented**: `Authorization: Bearer <supabase-access-token>` â€” a real Supabase Auth JWT (not a bespoke scheme). The route calls `supabase.auth.getUser(token)` against the anon client to validate it, then authorizes the resulting user as either (a) the owning merchant (`merchant.auth_user_id === user.id`) or (b) any lender (a row in `lenders` with `auth_user_id === user.id`). This was the simplest correct option given what actually exists: milestone 3's signup UI persists no client-side session (per its handoff seam) and milestone 12 (lender portal + its auth) doesn't exist yet, so there was no session/JWT plumbing to reuse and building a parallel bespoke auth layer would just be thrown away by milestone 12. The aggregate query itself runs on the **service-role client**, bypassing RLS â€” `transactions`' RLS policy (`data-model.md`) only has an owning-merchant `select` policy, no lender policy, so RLS alone can't authorize a lender's read of someone else's transactions; the route's bearer-token check above is what actually gates access, RLS is not relied on here. **Flag for milestone 12**: once lenders get a real portal, this route's "any lender can view any merchant's revenue" check matches `data-model.md`'s stated RLS intent for `merchants`/`reports` ("lenders can select any merchant/report row â€” search is core to the product"), so no narrowing is expected to be needed, but confirm that's still the intent before this milestone's auth check is reused/extended.
- **New index**: `idx_transactions_merchant_created` on `transactions (merchant_id, created_at)` in [supabase/migrations/0002_revenue_indexes.sql](supabase/migrations/0002_revenue_indexes.sql), covering exactly the filter (`merchant_id`) and sort/bucket (`created_at`) the route's query does. **Applied to the live Supabase project** â€” the agent's sandboxed environment could only reach the network over HTTPS (raw TCP to port 5432 was blocked, so `psql` couldn't reach `db.<project>.supabase.co` directly), so the user ran this migration file manually against the live database after it was written.
- **Computation approach**: plain in-route aggregation, not a SQL view or scheduled function. The route does one indexed `select(amount, created_at)` query per request and sums/buckets in JS. Given the 2-day timeline and current data volume (single digits to low hundreds of transactions per merchant), this is simpler to get right and to change later than maintaining SQL views that would need to stay in sync with any future column changes; revisit only if a merchant's transaction count grows large enough that summing in JS becomes the bottleneck (not the case today).
- **Trend bucketing**: `trend: [{ period, amount }]`, `period` = `YYYY-MM-DD` by default, or `YYYY-MM` via `?granularity=monthly` (additive query param, not in the frozen `api-contracts.md` shape â€” omit it for the documented default). Milestone 7's chart can request whichever granularity it needs.
- **Verified against real data**: live-tested locally (Next dev server + real Supabase project, not a local/mocked DB) against the real "suya joint" merchant (`27608236-61e7-4a96-afdc-e2d3d872af5c`) and its real milestone-5 sandbox transaction (â‚¦30,000 from a live Monnify Bank Simulator transfer) â€” `grossInflow`/`verifiedRevenue` both returned `30000` correctly. Seeded three additional `transactions` rows directly (clearly marked: `monnify_reference` prefixed `TEST-M6-SEED-`, `raw_payload: {"test": true}`) spanning two different months to exercise daily and monthly trend bucketing, confirmed both granularities summed correctly (`54000` gross across 4 rows, correct per-day and per-month buckets), then deleted all three seeded rows and reconfirmed `grossInflow` returned to `30000`. Also confirmed `401` for missing/garbage bearer tokens and `403` for a valid token belonging to an unrelated merchant. Also re-verified against the live Render deployment after pushing â€” see below for the transcript.
- **Seam left for milestone 7** (revenue dashboard): call `GET /api/merchants/:id/revenue` with the merchant's own Supabase session access token (obtained via `supabase.auth.signInWithPassword` once milestone 3/7 wires up a real signupâ†’session flow â€” currently no client-side session exists, per milestone 3's seam) to render the totals/trend chart. `?granularity=monthly` is available if the chart wants a coarser view than daily.

## Milestone 8 â€” Fraud rule engine

- `lib/fraud.ts`: four standalone, pure rule functions (`checkCircularTransfers`, `checkSelfFunding`, `checkIdenticalTransfers`, `checkVelocitySpike`) plus a `runFraudChecks` orchestrator that queries the relevant Supabase windows and writes `fraud_flags` rows. Wired into `app/api/webhooks/monnify/route.ts` right after the transaction insert, before acking, per `architecture.md`'s diagram. `GET /api/merchants/:id/revenue` now filters `verifiedRevenue` by open flags. Full request/response/field detail in [api.md](api.md).
- **`payer_account` null handling** (applies to circular_transfer, self_funding, identical_transfers â€” all group/match by payer): a null `payer_account` (Monnify omitted `paymentSourceInformation`) is excluded from same-payer grouping entirely â€” it can't trigger or contribute to a same-payer match. This is a deliberate under-detection tradeoff: matching null-to-null across unrelated payers would produce false positives. `velocity_spike` is unaffected since it doesn't group by payer.
- **Self-funding's real limitation**: `checkSelfFunding` compares `payer_account` against a `merchantPersonalAccountNumber` parameter, but `merchants` has no such column â€” milestone 2's KYC mock only ever stored a verified boolean + hashed `kyc_reference`, never a raw BVN/NIN or personal account number (see milestone 2/4 entries, [[kyc-stretch-goal]]). The webhook route calls `runFraudChecks(supabase, inserted, null)` â€” always `null` today, so this rule is a correct, documented no-op against real data, not a stub. It's implemented against a real (if currently absent) identity value rather than a fabricated one, so it starts working the moment such a value is captured (e.g. if a future milestone adds a `merchants.personal_account_number` column), with no rule-logic change needed.
- **Flag idempotency**: two layers. (1) `runFraudChecks` fetches existing `fraud_flags` for the transaction and skips any `rule_type` already present before inserting. (2) `supabase/migrations/0004_fraud_flags_indexes.sql` adds a unique index on `(transaction_id, rule_type)` as a DB-level backstop, plus an index on `(status, transaction_id)` for the revenue route's flag lookup â€” **not yet applied to the live Supabase project** (same raw-Postgres-TCP/DNS limitation as migrations 0002/0003; user needs to apply it manually).
- **`verifiedRevenue` definition landed as**: gross sum minus any transaction with *any* open `fraud_flags` row, regardless of severity â€” not hardcoded to a high/medium severity list. Since all four rules are high or medium (per `fraud-rules.md`), this is equivalent to milestone 6's proposed "open, high or medium" threshold today, but stays correct without a code change if a future rule is ever added at low severity and deliberately excluded from the filter. Overridden flags (`status: "overridden"`) do not exclude a transaction â€” confirmed live (see below).
- **Verification â€” rules 1 (circular_transfer) and 3 (identical_transfers)**: verified via the real webhook path â€” synthetic but correctly HMAC-signed (`HMAC-SHA512(MONNIFY_SECRET_KEY, rawBody)`) payloads POSTed to the local dev server's `/api/webhooks/monnify` against the real "suya joint" merchant (`27608236-61e7-4a96-afdc-e2d3d872af5c`, account `4003488430`), exercising the actual production code path end-to-end (signature verification â†’ merchant match â†’ insert â†’ fraud engine â†’ flag insert). Trigger cases (3 same-payer txns; 5 identical-amount same-payer txns) produced the expected flag; no-trigger cases (2 same-payer; 4 identical-amount same-payer) did not. `monnify_reference` prefixed `TEST-M8-SEED-*` per the milestone 6 convention; all test transactions and flags deleted afterward, confirmed the merchant's only remaining transaction is the real milestone-5 sandbox payment (â‚¦30,000) and `fraud_flags` is empty again.
- **Verification â€” rule 4 (velocity_spike)**: first attempted against "suya joint" via the real webhook path with a directly-seeded (backdated `created_at`, bypassing the webhook since it always uses `now()`) 60-row baseline â€” but rules 1/3's own test transactions from moments earlier were still inside the rolling 1h window and inflated the count, confounding a clean no-trigger case (not a bug â€” `velocity_spike` correctly counts *all* recent transactions merchant-wide, not per-payer, per `fraud-rules.md`'s SQL sketch). ## Milestone 21 â€” Outcome-tracking infrastructure

- **Why this milestone exists**: the user explicitly asked for this after I flagged that recalibrating the credit score / loan recommendation / loan terms against real repayment outcomes is "blocked structurally" â€” every loan today is milestone 15's simulated deduction, not a real disbursement, so no real outcome exists yet. Rather than build nothing (and lose the chance to capture prediction data as it happens) or fabricate a recalibration with nothing real to calibrate against, this milestone builds the **data-capture pipe only** â€” it does not change how any score or recommendation is computed.
- **New migration**: `supabase/migrations/0009_loan_outcome_tracking.sql` adds `loans.credit_score_at_approval`/`loans.recommended_loan_amount_at_approval` â€” a snapshot of what the model predicted at the exact moment a loan was approved, captured in the same `POST /api/loans/:id/approve` query milestone 20 added (no extra query needed â€” same `reports` row already fetched for the credit-score tier lookup).
- **New endpoint**: `GET /api/admin/loan-outcomes` (admin-secret-gated, same pattern as milestone 14's fraud queue) lists every loan with its prediction snapshot alongside a **derived** (never stored/cached) `outcome`: `"not_yet_approved"` (still `pending`), `"in_progress"` (approved/repaying, no period overdue), `"delinquent"` (at least one unpaid period past its `dueDate`), `"repaid_full"` (`status: "repaid"`). - **`amountRecommendedDelta`**: `amount - recommendedLoanAmountAtApproval`, included per loan â€” how far the lender's actual decision diverged from what the model suggested. This is the exact pairing (predicted vs. actual lender decision, and eventually predicted vs. actual repayment outcome) a future recalibration pass would query; building the table/endpoint now means that's a query away later instead of a data-modeling exercise started from scratch when real outcomes exist.
- **Verified live**: seeded the same `TEST-M20-SEED-*` merchant/loan used for milestone 20's verification (single combined live pass, both milestones' behavior checked together since they share the same approval flow) â€” confirmed `GET /api/admin/loan-outcomes` included the freshly-approved loan with `outcome: "in_progress"` and `predicted.creditScoreAtApproval`/`predicted.recommendedLoanAmountAtApproval` exactly matching the values captured at approval. - **No real secrets committed**: only the approve route, the new admin route, the new migration, and this doc set changed; `.env` untouched.

## Milestone 22 â€” Public credit-lookup API (Phase 4 v1)

- **Why this milestone exists**: the user asked to scope Phase 4 ("portable, cross-platform identity") for a first version, and chose the "public read API" option over a merchant-owned exportable credential or leaving it as positioning-only â€” the "Stripe for credit scoring" framing from the original strategy discussion made concrete: a third-party platform can query a merchant's score without that merchant needing an existing lender-portal relationship with them.
- **New auth layer, deliberately separate from `lib/lender-auth.ts`**: `lib/public-api-auth.ts`'s `authenticateApiClient` checks an `x-api-key` header (SHA-256-hashed, compared against `api_clients.api_key_hash`) â€” no Supabase session, since callers are external platforms, not PROOFR lenders or merchants. **No public self-serve signup** â€” same posture milestone 12 established for lenders. `scripts/provision-api-client.ts` is a real, committed operational tool (unlike the disposable test-seeding scripts elsewhere in this codebase) that generates a raw key, prints it exactly once, and stores only its hash.
- **New migration**: `supabase/migrations/0010_public_api_clients.sql` â€” `api_clients` (provisioned platforms) and `api_access_log` (every query, since no rate-limiting exists yet â€” this is the only abuse-detection mechanism today). - **New endpoint**: `GET /api/public/score?phone=<E.164>` â€” looks up a merchant by exact `phone` match, **requires `approval_status: "approved"`** (an unapproved/rejected merchant's existence isn't disclosed â€” a 404 either way, same non-disclosure stance milestone 12 established for search), and returns exactly three summary fields (`confidenceScore`, `creditScore`, `recommendedLoanAmount`) plus `scoredAt` â€” deliberately **less** than what an authenticated lender sees (no revenue figures, no score breakdowns, no fraud flags, no transaction data), since a third party with no vetted relationship to this specific merchant shouldn't get more than a lender the merchant's own report was actually shared with.

## Milestone 23 â€” Public API consent gate

- **Why this milestone exists**: immediately after milestone 22 shipped, I flagged its consent gap directly rather than letting it sit in a doc, and the user's next message was "fix the gap." This milestone closes it â€” not a redesign of milestone 22, an additive gate on top of it.
- **Design choice: opt-in, not opt-out**: a merchant is invisible to `GET /api/public/score` by default, including every merchant who existed before this migration ran (`merchants.public_api_consent_at` defaults to `null` for everyone â€” nobody was silently grandfathered into being queryable). This was a deliberate choice over the alternative (visible by default, with an opt-out toggle) â€” a merchant never agreed to third-party exposure when they originally signed up for proof-of-revenue/lender-portal features, so the correct default for a brand-new exposure surface is "not exposed until asked," not "exposed until they notice and disable it."
- **New migration**: `supabase/migrations/0011_public_api_consent.sql` adds `merchants.public_api_consent_at` (timestamptz, nullable) â€” the timestamp itself (not just a boolean) records *when* consent was granted, useful if this is ever audited.
- **New endpoint**: `GET`/`POST /api/merchants/:id/public-api-consent` â€” merchant-owner-only auth (same bearer-token-owns-the-record pattern used throughout, e.g. `POST /api/merchants/:id/report`), no lender path since this is squarely the merchant's own decision. `POST { consent: true }` sets the timestamp to now; `POST { consent: false }` sets it back to `null` â€” full revocation, not a soft flag.
- **`GET /api/public/score` updated**: now filters on `public_api_consent_at` being non-null in addition to `approval_status: "approved"`. An approved-but-unconsented merchant gets the exact same `404` a nonexistent phone number would â€” preserves the existing non-disclosure stance (a caller still can't tell "doesn't exist" from "not approved" from "hasn't consented").
- **UI**: `app/dashboard/page.tsx` gained a "Third-party credit lookups" card with a plain-language explanation of what consenting actually exposes (score + recommended amount only, never revenue/transaction detail â€” same framing as the endpoint's own response-shape limits) and an on/off toggle reflecting live state, fetched alongside the existing revenue/flags data on load.
- **Verified live, the full cycle, not just the happy path**: seeded a `TEST-M23-SEED-*` merchant (approved, with a real report) â€” confirmed default consent state is `{ consentGranted: false, consentedAt: null }`; confirmed `GET /api/public/score` returned `404` before any consent action; granted consent, confirmed the identical query immediately returned `200` with correct data; revoked consent, confirmed the identical query immediately returned `404` again; confirmed a second, unrelated merchant's valid bearer token gets `403` attempting to change the first merchant's consent (not just "isn't provided a UI for it" â€” actually rejected server-side). All seeded data (both merchants, transaction, report, both auth users) and the test `api_clients` row deleted afterward.
- **No real secrets committed**: only the new consent route, the updated public-score route, the dashboard page, the new migration, and this doc set changed; `.env` untouched.
- **Still open, not bundled into this fix**: a merchant has no visibility into *who* has queried them or how many times â€” `api_access_log` already has the data, it's just not surfaced to the merchant yet. Rate-limiting (noted in milestone 22's seam) is also still unbuilt â€” consent controls *whether* lookups are possible, not how many a compromised key could make.
