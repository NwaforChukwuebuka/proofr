# PROOFR — Handoff Log

Cross-milestone continuity: what shipped, what's mocked and why, env vars added, and what the next milestone's dev (human or agent) needs to know before touching this code. Append one entry per milestone as it completes — never rewrite past entries, only add to them if a later milestone changes something they assumed.

## Convention: visual theme

Every **frontend** milestone (3, 7, 9, 11, 13, 14) must read every image in `ui-inspirations-theme/` before writing any UI — it's the visual spec (Moniepoint-style Nigerian fintech look: brand-blue backgrounds, white rounded cards, bold headlines, pill buttons, balance-card motifs; see `app/globals.css`'s `--brand`/`--brand-dark`/`--brand-tint` vars from milestone 3 for the established palette). Milestone 3's first pass missed this folder (it isn't referenced from `plan.md`/`architecture.md`) and shipped a generic black/white UI before being redirected — don't repeat that.

## Convention: `integration.md`

After every **frontend** milestone (3, 7, 9, 11, 13, 14), the frontend dev drops an `integration.md` in the project root (overwrite the previous one — it reflects current state, not history; this log is the history) covering:
- Which backend endpoints from [api.md](api.md) it actually calls, and with what shapes.
- Any mismatch found between `api.md`/`api-contracts.md` and what the API really returned, and how it was resolved (fixed the backend, or adapted the frontend — say which).
- Any client-side assumption baked in (e.g. polling vs. realtime, optimistic UI, error states) that a later milestone should know about.
- Manual test notes: what was clicked through in a real browser against the deployed Render URL, per this project's standing rule to test UI changes live before calling them done.

This entry in `handoff.md` should then get a one-line pointer to that milestone's `integration.md` findings, once available.

## Milestone 6 — Revenue engine

- `GET /api/merchants/:id/revenue` implemented in [app/api/merchants/[id]/revenue/route.ts](app/api/merchants/[id]/revenue/route.ts). Full request/response/auth detail in [api.md](api.md).
- **"Gross inflow" vs "verified revenue" definition** (PRD names both, doesn't define the gap): for this milestone they are **identical** — both are `sum(transactions.amount)` for the merchant. Reasoning: "verified" is meant to mean "with fraud-flagged activity excluded," but the fraud rule engine (milestone 8) doesn't exist yet and `fraud_flags` is empty, so there is nothing to exclude today. Making `verifiedRevenue` a different number now (e.g. some ad-hoc discount) would just be an arbitrary number, not a real verification signal. **Milestone 8 must revisit this**: once fraud rules populate `fraud_flags`, `verifiedRevenue` should become `sum(amount) for transactions with no open high/medium-severity flag` (exact threshold is milestone 8's call, see `fraud-rules.md`) while `grossInflow` stays the unfiltered sum — that's the real distinction the PRD is gesturing at.
- **Gross vs net `amount` decision** (seam flagged by milestone 5): both `grossInflow` and `verifiedRevenue` use the first-class `transactions.amount` column (Monnify's gross `amountPaid`), not the fee-adjusted `settlementAmount` buried in `raw_payload->'eventData'->>'settlementAmount'`. Reasoning: `amount` is indexed, typed, and always present; `settlementAmount` is an unindexed JSON path that would need a migration to promote to a first-class column to aggregate cheaply and correctly, and Monnify's fee is a processing-cost detail, not a fraud/authenticity signal — conflating "fee-adjusted" with "verified" would misuse the word the PRD is careful to use for fraud screening. If a later milestone wants a net/settlement figure (e.g. "what actually lands in the merchant's bank"), it should be a third, separately-named field, not a redefinition of `verifiedRevenue`.
- **Auth implemented**: `Authorization: Bearer <supabase-access-token>` — a real Supabase Auth JWT (not a bespoke scheme). The route calls `supabase.auth.getUser(token)` against the anon client to validate it, then authorizes the resulting user as either (a) the owning merchant (`merchant.auth_user_id === user.id`) or (b) any lender (a row in `lenders` with `auth_user_id === user.id`). This was the simplest correct option given what actually exists: milestone 3's signup UI persists no client-side session (per its handoff seam) and milestone 12 (lender portal + its auth) doesn't exist yet, so there was no session/JWT plumbing to reuse and building a parallel bespoke auth layer would just be thrown away by milestone 12. The aggregate query itself runs on the **service-role client**, bypassing RLS — `transactions`' RLS policy (`data-model.md`) only has an owning-merchant `select` policy, no lender policy, so RLS alone can't authorize a lender's read of someone else's transactions; the route's bearer-token check above is what actually gates access, RLS is not relied on here. **Flag for milestone 12**: once lenders get a real portal, this route's "any lender can view any merchant's revenue" check matches `data-model.md`'s stated RLS intent for `merchants`/`reports` ("lenders can select any merchant/report row — search is core to the product"), so no narrowing is expected to be needed, but confirm that's still the intent before this milestone's auth check is reused/extended.
- **New index**: `idx_transactions_merchant_created` on `transactions (merchant_id, created_at)` in [supabase/migrations/0002_revenue_indexes.sql](supabase/migrations/0002_revenue_indexes.sql), covering exactly the filter (`merchant_id`) and sort/bucket (`created_at`) the route's query does. **Applied to the live Supabase project** — the agent's sandboxed environment could only reach the network over HTTPS (raw TCP to port 5432 was blocked, so `psql` couldn't reach `db.<project>.supabase.co` directly), so the user ran this migration file manually against the live database after it was written.
- **Computation approach**: plain in-route aggregation, not a SQL view or scheduled function. The route does one indexed `select(amount, created_at)` query per request and sums/buckets in JS. Given the 2-day timeline and current data volume (single digits to low hundreds of transactions per merchant), this is simpler to get right and to change later than maintaining SQL views that would need to stay in sync with any future column changes; revisit only if a merchant's transaction count grows large enough that summing in JS becomes the bottleneck (not the case today).
- **Trend bucketing**: `trend: [{ period, amount }]`, `period` = `YYYY-MM-DD` by default, or `YYYY-MM` via `?granularity=monthly` (additive query param, not in the frozen `api-contracts.md` shape — omit it for the documented default). Milestone 7's chart can request whichever granularity it needs.
- **Verified against real data**: live-tested locally (Next dev server + real Supabase project, not a local/mocked DB) against the real "suya joint" merchant (`27608236-61e7-4a96-afdc-e2d3d872af5c`) and its real milestone-5 sandbox transaction (₦30,000 from a live Monnify Bank Simulator transfer) — `grossInflow`/`verifiedRevenue` both returned `30000` correctly. Seeded three additional `transactions` rows directly (clearly marked: `monnify_reference` prefixed `TEST-M6-SEED-`, `raw_payload: {"test": true}`) spanning two different months to exercise daily and monthly trend bucketing, confirmed both granularities summed correctly (`54000` gross across 4 rows, correct per-day and per-month buckets), then deleted all three seeded rows and reconfirmed `grossInflow` returned to `30000`. Also confirmed `401` for missing/garbage bearer tokens and `403` for a valid token belonging to an unrelated merchant. Also re-verified against the live Render deployment after pushing — see below for the transcript.
- **Seam left for milestone 7** (revenue dashboard): call `GET /api/merchants/:id/revenue` with the merchant's own Supabase session access token (obtained via `supabase.auth.signInWithPassword` once milestone 3/7 wires up a real signup→session flow — currently no client-side session exists, per milestone 3's seam) to render the totals/trend chart. `?granularity=monthly` is available if the chart wants a coarser view than daily.
- **Seam left for milestone 8** (fraud rule engine): once `fraud_flags` has real rows, come back to this route and change `verifiedRevenue`'s computation to exclude flagged transactions per the definition above — everything else in this route (auth, gross inflow, trend) should be unaffected.

## Milestone 8 — Fraud rule engine

- `lib/fraud.ts`: four standalone, pure rule functions (`checkCircularTransfers`, `checkSelfFunding`, `checkIdenticalTransfers`, `checkVelocitySpike`) plus a `runFraudChecks` orchestrator that queries the relevant Supabase windows and writes `fraud_flags` rows. Wired into `app/api/webhooks/monnify/route.ts` right after the transaction insert, before acking, per `architecture.md`'s diagram. `GET /api/merchants/:id/revenue` now filters `verifiedRevenue` by open flags. Full request/response/field detail in [api.md](api.md).
- **`payer_account` null handling** (applies to circular_transfer, self_funding, identical_transfers — all group/match by payer): a null `payer_account` (Monnify omitted `paymentSourceInformation`) is excluded from same-payer grouping entirely — it can't trigger or contribute to a same-payer match. This is a deliberate under-detection tradeoff: matching null-to-null across unrelated payers would produce false positives. `velocity_spike` is unaffected since it doesn't group by payer.
- **Self-funding's real limitation**: `checkSelfFunding` compares `payer_account` against a `merchantPersonalAccountNumber` parameter, but `merchants` has no such column — milestone 2's KYC mock only ever stored a verified boolean + hashed `kyc_reference`, never a raw BVN/NIN or personal account number (see milestone 2/4 entries, [[kyc-stretch-goal]]). The webhook route calls `runFraudChecks(supabase, inserted, null)` — always `null` today, so this rule is a correct, documented no-op against real data, not a stub. It's implemented against a real (if currently absent) identity value rather than a fabricated one, so it starts working the moment such a value is captured (e.g. if a future milestone adds a `merchants.personal_account_number` column), with no rule-logic change needed.
- **Flag idempotency**: two layers. (1) `runFraudChecks` fetches existing `fraud_flags` for the transaction and skips any `rule_type` already present before inserting. (2) `supabase/migrations/0004_fraud_flags_indexes.sql` adds a unique index on `(transaction_id, rule_type)` as a DB-level backstop, plus an index on `(status, transaction_id)` for the revenue route's flag lookup — **not yet applied to the live Supabase project** (same raw-Postgres-TCP/DNS limitation as migrations 0002/0003; user needs to apply it manually).
- **`verifiedRevenue` definition landed as**: gross sum minus any transaction with *any* open `fraud_flags` row, regardless of severity — not hardcoded to a high/medium severity list. Since all four rules are high or medium (per `fraud-rules.md`), this is equivalent to milestone 6's proposed "open, high or medium" threshold today, but stays correct without a code change if a future rule is ever added at low severity and deliberately excluded from the filter. Overridden flags (`status: "overridden"`) do not exclude a transaction — confirmed live (see below).
- **Verification — rules 1 (circular_transfer) and 3 (identical_transfers)**: verified via the real webhook path — synthetic but correctly HMAC-signed (`HMAC-SHA512(MONNIFY_SECRET_KEY, rawBody)`) payloads POSTed to the local dev server's `/api/webhooks/monnify` against the real "suya joint" merchant (`27608236-61e7-4a96-afdc-e2d3d872af5c`, account `4003488430`), exercising the actual production code path end-to-end (signature verification → merchant match → insert → fraud engine → flag insert). Trigger cases (3 same-payer txns; 5 identical-amount same-payer txns) produced the expected flag; no-trigger cases (2 same-payer; 4 identical-amount same-payer) did not. `monnify_reference` prefixed `TEST-M8-SEED-*` per the milestone 6 convention; all test transactions and flags deleted afterward, confirmed the merchant's only remaining transaction is the real milestone-5 sandbox payment (₦30,000) and `fraud_flags` is empty again.
- **Verification — rule 4 (velocity_spike)**: first attempted against "suya joint" via the real webhook path with a directly-seeded (backdated `created_at`, bypassing the webhook since it always uses `now()`) 60-row baseline — but rules 1/3's own test transactions from moments earlier were still inside the rolling 1h window and inflated the count, confounding a clean no-trigger case (not a bug — `velocity_spike` correctly counts *all* recent transactions merchant-wide, not per-payer, per `fraud-rules.md`'s SQL sketch). Re-verified cleanly against a disposable fixture merchant (created via `POST /api/merchants`, no cross-contamination) with the same backdated-baseline approach, this time seeding the "current" transaction directly and calling `runFraudChecks` directly rather than through the webhook (the fixture has no real Monnify account, so the webhook's merchant-matching step doesn't apply) — 1 recent transaction against the 60-row baseline did not trigger, a 2nd did (recent count exceeded 3x the baseline hourly average), confirming both directions.
