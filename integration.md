# Milestone 13 — Integration Notes (Lender portal UI)

Reflects current state as of this milestone. Overwrites milestone 11's `integration.md` per `handoff.md`'s convention — that milestone's history now lives in `handoff.md` only.

## What was built

- `app/login/page.tsx` — extended, not duplicated: after `supabase.auth.signInWithPassword`, the page checks for a `merchants` row matching the signed-in user first, then a `lenders` row, and routes to `/dashboard` or `/lender` accordingly. Neither match → an inline error ("This account isn't set up as a merchant or lender yet.") rather than a silent redirect. No second login form was built — same sign-in mechanism, one branch after success, per this milestone's build prompt.
- `app/lender/page.tsx` — lender landing + search. On load: session check → RLS-scoped read of the caller's own `lenders` row (`lenders_select_own` policy) to get `org_name` and to gate the page — a merchant session (or any session with no `lenders` row) is redirected to `/dashboard`, not shown lender UI. Search box calls `GET /api/lenders/search?query=`, renders each result's `businessName` and a score badge; `confidenceScore: null` renders a distinct "Not yet scored" pill (zinc, not red/green), never `0` or blank.
- `app/lender/merchants/[id]/page.tsx` — merchant summary + mock loan approval. Same lender-session guard as the landing page. Fetches the merchant's `business_name`/`approval_status` via a direct RLS-scoped `merchants` read (lenders can select any merchant row per `data-model.md`'s RLS policy — no new endpoint needed for the header), then `GET /api/lenders/merchants/:id` (bearer token) for the report data. A `404` from that call (merchant has no report yet) renders a plain "hasn't generated a Proof-of-Revenue report yet" message instead of erroring or hiding the page — the loan-approval form still renders below it, since loan approval doesn't require a report to exist. When a report does exist, the confidence score, verified/gross revenue, open fraud flags (reusing `lib/fraud-labels.tsx`'s `RULE_LABELS`/`SeverityBadge`, same as the merchant-facing report page), and a link to `/report/:merchantId?reportId=<reportId>` (milestone 11's existing unauthenticated share-link view — not a second report renderer) are shown.
- **Loan approval is one action, not two steps**: an amount input + "Approve mock loan" button calls `POST /api/loans` then immediately `POST /api/loans/:id/approve` in sequence, and renders the resulting `mockRepaymentSchedule` under an "Estimated repayment schedule" heading with an explicit caption ("A rough placeholder — an even split with no interest/fees modeled, not real amortization") per milestone 12's handoff note not to oversell the 3-way split as real amortization logic.

## Endpoints called, and shapes

- `GET /api/lenders/search?query=` (lender bearer token) — as documented in `api.md`; `confidenceScore` rendered as `null` → "Not yet scored" badge, otherwise a numeric badge colored by tone (green ≥80, amber ≥50, red below).
- `GET /api/lenders/merchants/:id` (lender bearer token) — identical shape to `GET /api/merchants/:id/report`'s bearer-token response, consumed as documented. `404` handled explicitly (see above), not treated as a generic error.
- `POST /api/loans` then `POST /api/loans/:id/approve` (lender bearer token) — exactly as documented in `api.md`'s milestone 12 section; no mismatch found.
- **No mismatch found between `api.md`/`api-contracts.md` and what the live routes actually returned** — all four routes matched their documented shapes exactly, including the `confidenceScore: null` case.

## Client-side assumptions

- **Lender-only gating is client-side, on top of backend auth.** `GET /api/lenders/merchants/:id` and the direct RLS `merchants` read both succeed for a merchant's own bearer token too (by design — see milestone 12's note that this route is intentionally merchant-or-lender). The UI closes that gap itself: both `/lender` and `/lender/merchants/:id` check for a `lenders` row on load and redirect a non-lender session to `/dashboard` before rendering any lender-only action (search, loan approval). `POST /api/loans` itself is still lender-only server-side (`authenticateAsLender`), so even a merchant who somehow reached the page couldn't approve a loan — the client-side redirect is a UX guard, not the only line of defense.
- **No polling/realtime** — search results and the merchant summary are fetched once per action (search submit, page load); no subscription, matching the read-then-act nature of the lender flow (no data here changes moment-to-moment the way a merchant's own revenue dashboard does).
- **Business name for the summary header comes from a direct `merchants` table read, not from the report response**, so a merchant with no report yet still gets a proper page header instead of an empty/generic one.
- **Loan approval has no separate "create" and "approve" UI step** — chosen over mirroring the two-call API shape 1:1 because a lender approving a loan in this hackathon's flow has no reason to leave one "pending" (no review queue, no other party who acts on a pending loan before approval) — see `api.md`'s milestone 12 section for the two-call shape this button drives.
